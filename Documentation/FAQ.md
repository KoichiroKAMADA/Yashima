# FAQ

This page answers the questions that usually come up when deciding whether
Yashima belongs in a Swift app.

## Is Yashima an image loading library?

No. Yashima does not download remote images, manage HTTP cache policy, prefetch
URLs, or provide image views.

Use Yashima when your app locally generates an image-like artifact, such as a
thumbnail, preview, chart, map snapshot, or rendered document page, and can
generate that artifact again if the cache is cleared.

For remote images, use an image pipeline such as
[Nuke](https://github.com/kean/Nuke) or
[Kingfisher](https://github.com/onevcat/Kingfisher).

## How is Yashima different from NSCache?

`NSCache` is a good in-memory cache for transient objects. Yashima adds a
file-backed layer, typed codecs, stable key + codec identity, async
get-or-generate APIs, and single-flight generation for concurrent misses.

Use `NSCache` when memory-only reuse is enough. Use Yashima when relaunches,
navigation, scrolling, or repeated local generation make file-backed reuse and
single-flight coordination useful.

## Can I use Yashima as a database?

No. Yashima is for disposable derived artifacts. It may remove entries during
cache clearing, trimming, corruption recovery, or app-level lifecycle work.

Do not store the only copy of user-created files, originals, documents,
recordings, or authoritative app data in Yashima. Use SwiftData, Core Data,
SQLite, GRDB, app-owned files, or another durable persistence layer instead.

## What belongs in a CacheKey?

The key should include every input that changes the generated bytes. For a
thumbnail, that might include an asset identifier, size, scale, crop mode,
appearance, locale, renderer version, and any content revision token.

If changing an input would change the artifact, put that input in the key or in
the codec identity. Avoid keys that depend on process-local values such as
Swift `hashValue`.

## Why is the codec part of cache identity?

The same `CacheKey` can produce different bytes when it is stored through
different codecs. A PNG thumbnail, a JPEG thumbnail, and compressed data should
not collide just because the logical artifact key is the same.

Yashima therefore treats effective cache identity as:

```text
CacheKey + CacheCodec.identifier
```

## What does single-flight mean?

When several callers miss the same key and codec at the same time, Yashima can
run one producer task and share the result with all waiters. This avoids doing
the same expensive local work repeatedly during fast scrolling, repeated screen
entry, or concurrent startup paths.

The default policy shares work. `YCache.Options.uiLifecycle` can also cancel the
producer when every UI waiter disappears.

## How should I measure whether Yashima helped?

Measure repeated local generation before and after adoption. Count how often the
producer runs for the same logical artifact, then check memory hits, storage
hits, and shared in-flight work after Yashima is introduced.

The most useful result is often "this thumbnail was generated once instead of
many times," not a claim that Yashima is faster than another cache tool.

See [Adoption Measurement](AdoptionMeasurement.md) for a practical checklist.

## When should I use `YCache.Options.uiLifecycle`?

Use `.uiLifecycle` for work whose value exists only while the current UI caller
is still interested, such as SwiftUI `List` cells, `LazyVGrid` tiles, and
`.task(id:)` driven previews.

Keep the default options for background work, detail screens, exports, and any
generation where finishing the producer still has value after the first caller
leaves.

## Does Yashima make disk I/O non-blocking?

No. Yashima is Swift Concurrency-first, but it still uses Foundation file I/O
under the hood. The API is designed for async workflows, actors, and `Sendable`
values, but it does not claim that disk access itself is fully non-blocking.

## Does Yashima store `nil` or failures?

No. Optional generators may return `nil`, and Yashima will share that `nil` with
current waiters, but it does not persist a negative cache entry.

Yashima also does not store thrown errors or cancelled producer results.
