# Comparison

This page compares Yashima with adjacent tools so developers can choose the
right cache for a specific job. It is not a ranking. The best choice depends on
who owns the data, whether the value can be regenerated, and whether the cache
needs a full image-loading pipeline.

Snapshot checked: 2026-07-06.

## Summary

| Tool | Best fit | Storage scope | Concurrency shape | Use instead of Yashima when |
|---|---|---|---|---|
| `NSCache` | In-memory transient objects | Memory only | Thread-safe object cache | You only need memory reuse and system-pressure eviction |
| `hyperoslo/Cache` | General Swift caching with memory and disk storage | Memory + disk | Library-managed cache APIs | You want a broader general-purpose cache abstraction |
| `pinterest/PINCache` | Established parallel object cache | Memory + disk | Parallel Objective-C era cache design | You need its mature object-cache behavior or legacy ecosystem fit |
| `kean/Nuke` | Image loading pipeline | Network, decoding, processing, memory, disk | Image request pipeline | You are downloading and displaying remote images |
| `onevcat/Kingfisher` | Web image downloading and caching | Network image cache | Image request pipeline | You need a full remote-image loading library |
| Yashima | Locally generated disposable artifacts | Memory + file-backed storage | Swift Concurrency get-or-generate with single-flight | Your app repeatedly regenerates thumbnails, previews, charts, summaries, or rendered documents |

## Yashima's Niche

Yashima is built for values that meet all of these conditions:

- The app generates the value locally.
- The value is expensive enough that regenerating it repeatedly hurts UX,
  battery, or launch/navigation latency.
- The value is safe to delete because the app can generate it again.
- The key can describe every input that changes the generated bytes.
- Concurrent misses for the same key should usually share work.

Examples include app-generated thumbnails, MapKit snapshots, chart images,
filter previews, waveforms, rendered summaries, compressed first-paint document
artifacts, and small metadata derived from local assets.

## When Other Tools Are Better

### NSCache

Apple describes `NSCache` as a mutable collection for transient key-value pairs
that may be evicted when resources are low. It is thread-safe and simple. It is
often the right first choice for in-memory object reuse.

Use `NSCache` instead when:

- memory-only reuse is enough;
- values do not need to survive app relaunch;
- the app already owns generation scheduling and cancellation;
- a small Foundation-only object cache is the clearest solution.

Use Yashima when you also need file-backed reuse, typed codecs, stable key +
codec identity, async get-or-generate APIs, and single-flight generation.

### Nuke and Kingfisher

Nuke and Kingfisher are image-loading libraries. Their center of gravity is
fetching images from the web, decoding them, processing them, caching them, and
displaying them in app UI.

Use Nuke or Kingfisher instead when:

- the primary input is a remote image URL;
- the app needs an image request pipeline;
- HTTP caching, progressive loading, prefetching, or view integration is central
  to the feature.

Yashima intentionally does not download images. It can cache a JPEG or PNG that
your app generated, but the producer belongs in your app or in a dedicated image
pipeline.

### hyperoslo/Cache and PINCache

`hyperoslo/Cache` and `pinterest/PINCache` are established general-purpose cache
libraries with their own design histories and ecosystems. They may be a better
fit when a project wants those broader cache abstractions or compatibility with
existing code built around them.

Yashima is narrower. It is designed around a small Swift Concurrency-first API,
typed codecs, disposable generated artifacts, and single-flight get-or-generate
workflows.

### Databases and Persistence Layers

Do not use Yashima as a database. If a value is authoritative, user-created, or
impossible to regenerate, store it in an appropriate persistence layer such as
SwiftData, Core Data, SQLite, GRDB, files owned by your app, or another durable
store.

Yashima values may be removed by lifecycle operations, trimming, corruption
recovery, or app-level cache clearing. That is a feature for disposable
artifacts and a bug for authoritative data.

## Feature Matrix

| Capability | Yashima | NSCache | Nuke | Kingfisher |
|---|---:|---:|---:|---:|
| Swift Concurrency-first local artifact API | Yes | No | No, image pipeline focused | No, image pipeline focused |
| File-backed cache layer | Yes | No | Yes | Yes |
| Remote image downloading | No | No | Yes | Yes |
| Typed codec model | Yes | No | Image/data pipeline focused | Image/data pipeline focused |
| Get-or-generate API | Yes | No | Request pipeline | Request pipeline |
| Single-flight generation for local artifacts | Yes | No | Pipeline-specific | Pipeline-specific |
| Cancellation policy for UI-generated local work | Yes | App-owned | Pipeline-specific | Pipeline-specific |
| Suitable for authoritative data | No | No | No | No |
