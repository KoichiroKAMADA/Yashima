# Designing Cache Keys

Yashima can make storage, memory, and concurrent generation predictable, but the
host app still owns the hardest cache question: what exactly changes the
generated bytes?

Start with a stable identity for the thing being rendered:

```swift
let key = CacheKey(namespace: "video-thumbnails", identity: videoIdentity)
```

Then add every input that can change the artifact:

```swift
let key = CacheKey(namespace: "video-thumbnails", identity: videoIdentity)
    .variant("second", thumbnailSecond)
    .variant("pixels", "\(pixelWidth)x\(pixelHeight)")
    .variant("scale", scale)
    .variant("appearance", appearance)
    .version("renderer", 1)
```

## Complete Keys

If two generations can produce different bytes, their ``CacheKey`` values should
be different. Include size, scale, locale, color scheme, renderer options,
source-data revision, and stable digests for large inputs.

Avoid Swift `hashValue` and `Hasher` for persisted cache identity because they
are not stable across process launches. If you need to summarize a large input,
use a stable digest such as SHA-256 before constructing the key.

## Namespaces

Use ``CacheKey/namespace`` to separate artifact kinds inside one shared
``YCache``:

```swift
let thumbnailKey = CacheKey(namespace: "video-thumbnails", identity: videoID)
let durationKey = CacheKey(namespace: "video-durations", identity: videoID)
```

Namespaces are logical partitions, not independent storage roots. For a typical
app, prefer one long-lived ``YCache`` instance and separate artifact kinds with
namespaces.

## Stable Identifiers

Use ``CacheKey/stableIdentifier`` when code outside Yashima needs a stable,
opaque, key-derived string for logs, auxiliary file names, or deduplication.
Do not treat it as a stored-entry identifier. Stored cache entries are identified
by both ``CacheKey`` and ``CacheCodec/identifier``.
